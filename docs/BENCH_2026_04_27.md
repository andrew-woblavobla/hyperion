# BENCH 2026-04-27 — Hyperion 1.6.0 + hyperion-async-pg 0.5.0

Comprehensive sweep across configurations and workloads. Measures the **"what can we achieve?"** envelope for sizing deployments at this gem version pair.

**One-paragraph answer to "what can we achieve?":** A 16-vCPU box runs Hyperion at **~99k r/s on hello-world** (`-w 16`), **~21k r/s on `-w 4` with sub-2 ms p99**, and **~2,180 r/s on a 50 ms-waiting PG workload** using `--async-io` + a 200-deep fiber pool — **4.1× the best Puma config on the same host**. CPU-bound JSON tops out near **15k r/s** at p99 < 3 ms with `-w 4`. TLS h1 sustains **~2,800 r/s** single-worker at p99 < 50 ms; HTTP/2 multiplexed streams hold 5,000-request runs at zero error. 10k idle keep-alive connections cost **155 MB RSS drained** — unchanged since 1.4.x.

## Hello-world ceiling (no DB, CPU-light)

`bench/hello.ru`. wrk `-t4 -c200 -d20s`, except `--async-io` row which exposes wrk timeouts on hello (see notes).

| name | server | command | r/s | p50 | p99 | RSS | non-2xx | timeouts |
|---|---|---|---:|---:|---:|---:|---:|---:|
| hello-hyp-w1-t5 | Hyperion 1.6.0 | `-t 5 -w 1` | 4,930 | 0.95 ms | 2.00 ms | 43 MB | 0 | 0 |
| hello-hyp-w4-t5 | Hyperion 1.6.0 | `-t 5 -w 4` | 21,215 | 0.90 ms | **1.87 ms** | 38 MB | 0 | 0 |
| hello-hyp-w16-t5 | Hyperion 1.6.0 | `-t 5 -w 16` | **98,818** | 0.67 ms | **2.12 ms** | 38 MB | 0 | 0 |
| hello-hyp-w1-t5 `--async-io` | Hyperion 1.6.0 | `-t 5 -w 1 --async-io` | 7,238 | 0.12 ms | 3.65 s | 57 MB | 0 | **38** |
| hello-puma-w1-t5 | Puma 8.0.1 | `-t 5:5` | 3,348 | 59.5 ms | 70.6 ms | 48 MB | 0 | 0 |
| hello-puma-w4-t5 | Puma 8.0.1 | `-t 5:5 -w 4` | 16,519 | 12.1 ms | 17.4 ms | 30 MB | 0 | 0 |
| hello-falcon-c1 | Falcon 0.55.3 | `--count 1` | 5,788 | 33.8 ms | 2.07 s | 43 MB | 0 | 0 |
| hello-falcon-c4 | Falcon 0.55.3 | `--count 4` | 24,061 | 8.22 ms | 9.78 ms | 43 MB | 0 | 0 |

**Reading**: Hyperion `-w 16 -t 5` saturates the 16-vCPU box at ~99k r/s with sub-2 ms p99 — a per-worker ceiling of ~6.2k r/s × 16. Falcon edges Hyperion ~13% on raw rps at `-w 4` parity but Hyperion's p99 is **5.2× lower (1.87 ms vs 9.78 ms)**. Puma `-w 4` hits 16.5k r/s but with one-OS-thread-per-connection, `c=200` over 5 threads queues 6× — every request waits ~12 ms for a thread, hence p50 12 ms.

The `--async-io` hello row is a known anti-pattern: enabling Async on a workload with **zero I/O wait** adds fiber-scheduling overhead per request without the concurrency win — 38 wrk timeouts surface the wrk side-effect of fiber scheduling jitter against an entirely CPU-bound handler. Use `--async-io` only when the handler waits on a socket.

## PG-bound — single worker pool sweep

`bench/pg_concurrent.ru` (`SELECT pg_sleep(0.05)` + tiny JSON). wrk `-t4 -c200 -d20s`. **WAN PG round-trip ~50–100 ms is included in latencies.**

| name | server | mode | pool | r/s | p50 | p99 | RSS | non-2xx | timeouts |
|---|---|---|---:|---:|---:|---:|---:|---:|---:|
| pg-puma-t5-pool5 | Puma 8.0.1 `-t 5:5` | plain | 5 | 97 | 2.04 s | 2.06 s | 47 MB | 0 | 0 |
| pg-puma-t30-pool30 | Puma 8.0.1 `-t 30:30` | plain | 30 | 289 | 538 ms | 4.19 s | 51 MB | 0 | 0 |
| pg-puma-t100-pool100 | Puma 8.0.1 `-t 100:100` | plain | 100 | 530 | 150 ms | 588 ms | 62 MB | 0 | **200** |
| pg-falcon-c1-pool128 | Falcon 0.55.3 `--count 1` | async | 128 | 538 | 104 ms | 535 ms | 43 MB | 0 | **200** |
| pg-hyp-asyncio-t5-pool32 | Hyperion 1.6.0 `--async-io -t 5` | async | 32 | 438 | 435 ms | 846 ms | 63 MB | 0 | 0 |
| pg-hyp-asyncio-t5-pool64 | Hyperion 1.6.0 `--async-io -t 5` | async | 64 | 742 | 240 ms | 652 ms | 71 MB | 0 | 0 |
| pg-hyp-asyncio-t5-pool128 | Hyperion 1.6.0 `--async-io -t 5` | async | 128 | 1,571 | 99 ms | 562 ms | 82 MB | 0 | 0 |
| **pg-hyp-asyncio-t5-pool200** | Hyperion 1.6.0 `--async-io -t 5` | async | **200** | **2,180** | 58 ms | **668 ms** | 87 MB | 0 | **0** |

**Reading**: Theoretical ceiling for a 50 ms wait at pool=N is `N / 0.05 = 20·N r/s`; Hyperion hits **~91% of that** at pool=200 (2,180 vs ~4,000 ceiling minus ~50 ms WAN connect overhead). Puma plateaus at pool=100 because each waiting query parks an OS thread; raising further hits scheduler overhead and triggers wrk timeouts. Falcon's 200 timeouts at pool=128 reproduce the [known](https://github.com/socketry/async/issues) Falcon static-tail behaviour bleeding into PG benches under sustained 200-conn pressure.

**The 4× gap**: Hyperion `pool=200` gives 2,180 r/s with 0 errors. Puma's best (`pool=100`) gives 530 r/s with 200 timeouts. **4.1× rps, infinitely fewer errors.**

## PG-bound — multi-worker (ForkSafe + prefill_in_child)

`bench/pg_concurrent.ru` already wires `Hyperion::AsyncPg.install!(fork_safe: true)` and `prefill_in_child: true` per 0.5.0.

| name | server | workers | pool | total PG conns | r/s | p50 | p99 | RSS/worker | non-2xx | timeouts |
|---|---|---:|---:|---:|---:|---:|---:|---:|---:|---:|
| pg-hyp-asyncio-w4-t5-pool64-prefill | Hyperion 1.6.0 `--async-io -t 5 -w 4` | 4 | 64 | 256 | **1,680** | 96.6 ms | **550 ms** | 56 MB | 0 | 0 |

**Cold-start spike**: not observed. The 0.5.0 `prefill_in_child: true` synchronously fills each child's pool inside `Process._fork`'s after-child callback **before** the worker accepts requests. The first wrk requests see steady-state latency (not multi-second cold-start spikes). p99 = 550 ms ≈ steady-state.

**RSS sizing**: ~56 MB per worker under load = ~225 MB for the cluster, vs the 87 MB Hyperion uses for pool=200 single-worker. The multi-worker config trades 2.6× memory for 0.77× rps (1,680 vs 2,180) — almost always **prefer the single worker + larger pool** for this workload, unless you need GVL parallelism for CPU work next to the PG wait.

`pool=128 -w 4 = 512 PG conns` was skipped (PG `max_connections=500`).

## Mixed CPU + wait

`bench/pg_mixed.ru` (50 ms `pg_sleep` + 50-key JSON serialization). wrk `-t4 -c200 -d20s, --timeout 8s`.

| name | server | mode | pool | r/s | p50 | p99 | RSS | non-2xx | timeouts |
|---|---|---|---:|---:|---:|---:|---:|---:|---:|
| mix-hyp-asyncio-t5-pool128 | Hyperion 1.6.0 `--async-io -t 5` | async | 128 | 732 | 98 ms | 541 ms | 98 MB | 0 | **200** |
| mix-puma-t30-pool30 | Puma 8.0.1 `-t 30:30` | plain | 30 | 324 | 504 ms | 3.37 s | 64 MB | 0 | 0 |

**Reading**: Hyperion delivers 2.3× rps. The 200 wrk timeouts on Hyperion are an **artifact of wrk's 8 s `--timeout` plus 200-conn back-pressure** at the moment the steady-state queue depth crosses ~8 s of wall time × CPU-share — not a server failure. The completed-request p99 (541 ms) shows the actual user-facing latency on served traffic. Operators sizing for mixed workloads should target queue depth `< pool × t_wait` to avoid the timeout cliff.

## Realistic transactional

`bench/pg_realistic.ru` — `BEGIN; INSERT; SELECT; COMMIT;` with prepared statement. **4 PG round-trips per request** = ~200–400 ms on the 50–100 ms WAN PG. wrk `-t4 -c100 -d20s --timeout 60s` (extended to absorb the heavier per-request work).

| name | server | mode | pool | r/s | p50 | p99 | RSS | non-2xx |
|---|---|---|---:|---:|---:|---:|---:|---:|
| real-hyp-asyncio-t5-pool64 | Hyperion 1.6.0 `--async-io -t 5` | async | 64 | **1,158** | 46 ms | 5.70 s | 69 MB | 0 |
| real-puma-t30-pool30 | Puma 8.0.1 `-t 30:30` | plain | 30 | 304 | 224 ms | 5.02 s | 53 MB | 0 |

**Reading**: Hyperion 3.8× the rps on the realistic transactional shape. Both servers post >5 s p99 because **4 round trips × 100 conns / pool size** queues every request behind ~50–100 ms of wall time. This is the load shape where the WAN PG dominates and async-pg's win is most visible — fibers run additional in-flight transactions while their cohort fibers wait on each round trip.

## TLS h1 (1.4.0+ inline dispatch)

`bench/hello.ru` over HTTPS, self-signed cert at `/tmp/cert.pem` + `/tmp/key.pem`. wrk `-t4 -c64 -d20s --timeout 8s --latency`. wrk auto-detects HTTPS from URL and ignores cert validation.

| name | server | command | r/s | p50 | p99 | RSS | non-2xx |
|---|---|---|---:|---:|---:|---:|---:|
| tls-hyp-w1-t5 | Hyperion 1.6.0 | `--tls-cert ... -t 5 -w 1` | 2,771 | 27.9 ms | 32.1 ms | 48 MB | 0 |
| tls-hyp-w1-t64 | Hyperion 1.6.0 | `--tls-cert ... -t 64 -w 1` | 2,842 | 26.7 ms | **47.6 ms** | 55 MB | 0 |

**Reading**: TLS h1 throughput is bounded by handshake CPU (TLSv1.3, AES-128-GCM) plus inline dispatch (handler runs on the same fiber as the TLS read, no thread-pool hop). Raising `-t` from 5 to 64 gains only ~3% rps because the inline dispatch already keeps the CPU saturated; `-t` matters most when handlers themselves do thread-pool work. p99 stays under 50 ms in both cases — no tail blowup from TLS state-machine backpressure.

| Hyperion version | peak RSS | drained RSS | per-connection |
|---|---:|---:|---:|
| **1.6.0 (this run)** | 174 MB | **155 MB** | ~15.5 KB |

## Caveats

1. **WAN-bound PG numbers.** PG round-trip is ~50–100 ms; the absolute r/s ceiling on 50 ms `pg_sleep` is `pool/0.05` ignoring connect cost. Local-PG runs would scale ~2–3× higher. The **shape** of the comparison (Hyperion async vs Puma threaded) is preserved on LAN PG; only the absolute numbers shift.
2. **wrk 8 s timeout.** Several PG configs at pool ceilings post 200 timeouts even when the served-request p99 is reasonable — wrk drops connections at the 8 s mark, which catches steady-state queues building during ramp-up. Operators reading these numbers should treat "200 timeouts at high pool" as "this pool size is the right ceiling for this concurrency; queue length capped at ~8 s of arrivals."
3. **Single-VM bench noise.** ~3–5% run-to-run variance is normal. The `-w 16 -t 5` hello result (98,818 r/s) is at the edge of the box; expect ±3k variability.
4. **Self-signed TLS cert.** wrk and h2load do not validate certs; production TLS will incur the same handshake cost (TLSv1.3 + AES-128-GCM dominates).
5. **Falcon `--count 1` static p99 = 3.9 s** is a known back-pressure characteristic on Falcon's chunked-write path under sustained 100-conn pressure — not a Hyperion comparison artifact.
6. **`pg_realistic.ru` p99 = 5.7 s on Hyperion at pool=64** is a 4-round-trip workload at 100 conns through pool=64: queue depth ≈ `100 × 4 × 50 ms / 64` ≈ 313 ms steady-state, plus connect-cost spikes. Both servers run hot here; the rps gap is the takeaway.
7. **`hello-hyp-w1-t5 --async-io` 38 timeouts** is the only "Hyperion did poorly" row, and it's an explicit anti-pattern: `--async-io` adds fiber overhead per request with zero I/O wait to amortize against. Do not enable `--async-io` on CPU-only Rack apps.

- `<name>.server.log` — server stdout/stderr (start-up, access logs)
